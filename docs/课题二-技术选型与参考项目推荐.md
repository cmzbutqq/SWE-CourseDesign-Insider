# 课题二技术选型与参考项目推荐

## 1. 课题目标概述

课题二要求开发一套一体化监控平台，用于云环境下 IT 系统的统一监控与运维辅助。根据题目描述，系统至少需要覆盖以下能力：

- 在每台主机部署 `Agent`
- 自动扫描并识别主机上的服务和组件
- 根据识别出的技术栈，采用不同采集方式
- 将采集结果上传到服务端
- 在监控台统一展示主机信息、利用率、技术栈指标和调用链

题目真正的难点不在于“画图表”，而在于：

- 主机资产自动发现
- 技术栈识别与分类
- 不同组件的差异化采集
- 多类监控数据的统一接入和展示

因此，最合理的实现策略不是从零开发整套监控体系，而是尽量复用成熟开源组件，只对题目中无法直接复用的部分进行自研，从而最小化开发工作量并提高项目完成度。

## 2. 总体建议：复用成熟底座，薄封装自研

推荐采用“`自研轻量统一 Agent + 复用成熟监控底座`”的方案。

核心原则如下：

- 能直接复用的，绝不重写
- 能通过对接现有接口实现的，绝不自己造协议
- 能通过开源 Agent、Exporter、Java Agent 解决的，绝不自己采集底层细节
- 自研部分只聚焦在“统一发现、统一编排、统一管理、统一展示入口”

建议将系统分为两部分：

### 2.1 复用部分

- `Prometheus`：负责指标采集与存储
- `Grafana`：负责时序图表与监控面板展示
- `SkyWalking`：负责调用链、服务拓扑与应用性能监控
- 各类现成 `Exporter`：负责 Nginx、MySQL、Redis、主机等指标暴露
- `Spring Boot Actuator + Micrometer`：负责 Spring Boot 服务指标暴露

### 2.2 自研部分

- 统一 `Agent`
- 服务扫描与技术栈识别模块
- Agent 与服务端通信模块
- 主机/服务/组件资产管理后台
- 监控总览页和统一入口页

换句话说，本课题不建议自研“监控引擎”，而建议自研“监控编排与统一门户”。

## 3. 现有技术与开源项目概述

下面按课题能力拆分已有技术和项目。

### 3.1 主机指标监控

主机层面的监控包括 CPU、内存、磁盘、网卡、TCP 连接、操作系统信息等。这类能力已经非常成熟，不适合自己从零实现完整监控链路。

#### Prometheus

`Prometheus` 是当前最主流的开源监控与时序指标系统之一，适合做统一指标采集中心。

适合复用的能力：

- 定时抓取指标
- 标签化组织指标
- 时序数据存储
- 告警规则定义
- 与 Grafana 集成展示

适用场景：

- 主机资源指标
- 中间件指标
- 应用指标

参考项目：

- [prometheus/prometheus](https://github.com/prometheus/prometheus)

#### Node Exporter

`Node Exporter` 是 Prometheus 生态中最经典的主机指标采集器，几乎可以直接满足题目中的主机信息与利用率采集需求。

适合复用的能力：

- CPU、内存、磁盘、文件系统
- 网络接口
- TCP 连接相关基础信息
- Linux 主机级硬件与系统指标

结论：

- 如果只考虑最小开发量，主机指标采集优先直接复用 `Node Exporter`
- 如果题目要求必须体现“自研 Agent”，则可以由自研 Agent 负责安装、调用、注册或封装 Node Exporter，而不是自己重新实现所有主机采集逻辑

### 3.2 通用采集 Agent

#### Telegraf

`Telegraf` 是一个插件化程度很高的采集 Agent，支持 metrics、logs 等多种数据采集，拥有大量现成插件。

适合借鉴的点：

- 插件式架构
- 不同输入源对应不同采集模块
- 统一调度与上报机制
- 配置驱动式扩展

适合“抄”的部分：

- Agent 分层设计
- 插件注册机制
- 输入/处理/输出链路设计

参考项目：

- [influxdata/telegraf](https://github.com/influxdata/telegraf)

结论：

- 不建议直接把 Telegraf 当成课题成品核心，因为课程设计中仍需要体现“自动发现 + 技术栈识别 + 统一监控台”
- 但非常适合借鉴其插件架构，作为自研 Agent 的设计参考

### 3.3 主机资产发现与软件识别

#### osquery

`osquery` 可以把主机信息、进程、端口、软件包、系统服务等抽象成类似数据库表的查询接口，非常适合做资产发现和环境识别。

适合借鉴的能力：

- 进程识别
- 端口识别
- 软件包识别
- 服务识别
- 主机资产清单

适合“抄”的思路：

- 主机上有哪些进程
- 某个端口对应哪个进程
- 某个进程的命令行参数和工作目录是什么
- 根据路径、端口、进程名、启动参数判断服务类型

参考项目：

- [osquery/osquery](https://github.com/osquery/osquery)

结论：

- 非常适合借鉴“自动发现”部分的设计思路
- 如果时间紧，可以把它作为辅助发现工具使用
- 如果要保证部署简单，也可以不直接集成 osquery，而是在自研 Agent 中实现它的简化版能力

### 3.4 实时监控与自动发现体验

#### Netdata

`Netdata` 是一个强调实时性和自动发现体验的监控工具，安装简单、界面完整，适合参考“开箱即用”的产品体验。

适合借鉴的点：

- 自动发现思路
- 主机级实时监控页面
- 多维指标可视化布局
- 低门槛部署体验

参考项目：

- [netdata/netdata](https://github.com/netdata/netdata)

结论：

- 适合作为页面设计与产品体验参考
- 不建议直接拿来作为课题唯一核心，因为它对“统一平台自研程度”支撑不足

### 3.5 调用链与应用性能监控

题目要求展示“服务的指标、调用链等信息”，这一部分最不适合自研。

#### SkyWalking

`SkyWalking` 是成熟的开源 APM 平台，尤其适合 Java 微服务与 Spring Boot 应用。

适合复用的能力：

- 分布式调用链
- 服务拓扑图
- 接口性能监控
- 慢调用分析
- Java Agent 自动埋点

参考项目：

- [apache/skywalking](https://github.com/apache/skywalking)
- [apache/skywalking-java](https://github.com/apache/skywalking-java)

结论：

- 如果课题演示环境中有 Spring Boot 服务，`SkyWalking` 是最优选择
- 对课程设计而言，它能以最小自研成本提供最有说服力的“调用链展示”效果

#### Pinpoint

`Pinpoint` 也是成熟的 APM 系统，偏 Java 生态。

参考项目：

- [pinpoint-apm/pinpoint](https://github.com/pinpoint-apm/pinpoint)

结论：

- 可以作为 SkyWalking 的备选
- 但综合社区活跃度、资料丰富度和课程项目可落地性，优先级通常低于 SkyWalking

### 3.6 一体化可观测平台

#### SigNoz

`SigNoz` 是基于 OpenTelemetry 的一体化观测平台，整合了 metrics、logs、traces。

参考项目：

- [SigNoz/signoz](https://github.com/SigNoz/signoz)

优点：

- 一体化程度高
- 支持指标、日志、链路统一观测
- 界面较现代

不足：

- 体系相对偏重
- 课程设计的部署、调试和理解成本更高
- 如果团队对 OpenTelemetry 不熟，集成成本不一定比 Prometheus + Grafana + SkyWalking 更低

结论：

- 适合做备选方案
- 不建议作为当前课题首选落地方案

## 4. 课题二推荐技术选型

综合题目要求、开发周期、课程设计工作量和演示效果，推荐以下技术选型。

### 4.1 总体架构选型

- 主机 Agent：`Go`
- 服务端后端：`Spring Boot`
- 前端：`Vue 3 + Element Plus + ECharts`
- 指标采集与存储：`Prometheus`
- 监控图表展示：`Grafana`
- 调用链与服务拓扑：`SkyWalking`
- 业务数据库：`MySQL`
- 缓存：可选 `Redis`

### 4.2 为什么这样选

#### Agent 使用 Go

原因：

- 单文件部署方便
- 资源占用低
- 适合做守护进程
- 采集系统信息、执行命令、扫描进程都比较方便
- 并发能力强，适合周期性扫描与采集

Go Agent 更适合承担：

- 主机扫描
- 进程/端口识别
- 技术栈判断
- 启动或管理本地 Exporter
- 把发现结果和元数据上报到平台

#### 平台后端使用 Spring Boot

原因：

- 开发快，课程项目资料多
- 适合做后台管理系统
- 易于实现资产管理、配置管理、Agent 注册、任务下发等接口
- 与 MySQL、Redis、前端管理台的组合成熟

Spring Boot 后端更适合承担：

- 主机与服务资产管理
- Agent 注册与心跳
- 采集配置下发
- 统一视图组织
- 用户权限与告警配置

#### 前端使用 Vue 3

原因：

- 中后台界面开发效率高
- 适合课程设计快速实现“平台感”
- 与 Element Plus 组合成熟

前端重点页面建议包括：

- 监控总览
- 主机列表
- 主机详情
- 服务列表
- 服务详情
- 技术栈分布
- 告警与异常页

## 5. 最小化自研工作量的推荐实现方案

这是本课题最重要的部分。

### 5.1 建议复用的组件

直接复用：

- `Prometheus`
- `Grafana`
- `SkyWalking`
- `Node Exporter`
- `Spring Boot Actuator`
- `Micrometer`

按需复用：

- `mysqld_exporter`
- `redis_exporter`
- `nginx-prometheus-exporter`

借鉴设计但不一定直接集成：

- `Telegraf`
- `osquery`
- `Netdata`

### 5.2 建议必须自研的部分

为了满足题目要求并体现项目特色，建议只自研以下模块：

- `统一 Agent`
- `服务识别模块`
- `采集策略匹配模块`
- `平台管理后台`
- `统一总览页面`

### 5.3 自研 Agent 的最小职责

不要让 Agent 负责复杂时序存储和图表展示，它只做以下事情：

1. 采集主机基础信息
2. 扫描进程、端口、软件包和服务
3. 识别技术栈
4. 判断该主机需要接入哪些现成 Exporter 或 APM Agent
5. 把识别结果与注册信息上报到服务端

换句话说，自研 Agent 的价值在于“识别和编排”，而不是“替代 Prometheus”。

### 5.4 组件级最小落地方式

#### 主机指标

方案：

- 直接复用 `Node Exporter`
- 由自研 Agent 统一完成安装检查、进程保活、地址注册

这样可以在保留“自研 Agent”存在感的同时，避免自己重写 CPU、内存、磁盘、网络采集逻辑。

#### Spring Boot 服务指标

方案：

- 在 Spring Boot 示例应用中接入 `Actuator + Micrometer`
- 暴露 Prometheus 指标接口

这样可以非常低成本地实现：

- 请求数
- 响应时间
- JVM 指标
- 会话数等基础指标

#### MySQL / Redis / Nginx 指标

方案：

- 直接接入对应 Exporter
- 由 Agent 识别本机存在的组件并自动注册

#### 调用链

方案：

- 直接复用 `SkyWalking`
- 演示环境选用一个或两个 Spring Boot 微服务样例
- 接入 SkyWalking Java Agent

这样就能在答辩时完整展示：

- 服务调用链
- 拓扑图
- 请求耗时
- 异常调用

## 6. 推荐支持的第一批技术栈

为了控制范围，建议第一版只支持以下对象：

- `Linux` 主机
- `Spring Boot`
- `Nginx`
- `MySQL`
- `Redis`

这是最合理的范围，原因如下：

- 足够覆盖主机、Web、中间件、数据库、微服务
- 开源生态成熟
- 演示效果好
- 文档与资料丰富
- 团队可以在有限时间内做完

不建议第一版就支持：

- Windows
- Kubernetes
- Docker 全量自动编排
- 日志全文检索平台
- 大规模告警编排系统

这些方向工作量大，且对课程设计加分有限。

## 8. 最终推荐方案

综合考虑“实现难度、完成度、演示效果、最小化自研工作量”，本课题最推荐的方案如下：

### 8.1 方案结论

采用：

- `Go 统一 Agent`
- `Spring Boot 管理后台`
- `Vue 3 监控门户`
- `Prometheus + Grafana` 负责指标
- `SkyWalking` 负责调用链
- `Node Exporter + 各类 Exporter + Actuator/Micrometer` 负责具体采集

### 8.2 复用与自研边界

复用：

- 监控采集底座
- 时序数据存储
- 图表展示
- 调用链平台
- 中间件指标采集器

自研：

- 统一 Agent
- 自动发现与技术栈识别
- 采集策略分发
- 统一资产管理后台
- 一体化监控总览页

### 8.3 为什么这是最佳方案

- 满足题目对 Agent、自动发现、分类采集、统一展示的要求
- 最大程度复用成熟组件，降低开发风险
- 项目结构清晰，答辩时容易说明“哪些是平台底座，哪些是本项目创新点”
- 工作量可控，适合课程设计周期
- 最终效果容易做得完整，不容易烂尾

## 9. 建议的后续工作

如果后续继续推进实现，建议按以下顺序进行：

1. 先搭建 `Prometheus + Grafana + SkyWalking` 演示环境
2. 准备一个 `Spring Boot` 示例微服务和一个 `Nginx + MySQL + Redis` 演示环境
3. 开发最小版 `Go Agent`，先完成主机扫描和服务识别
4. 开发 `Spring Boot` 后台，实现主机注册、服务清单、策略配置
5. 开发 `Vue 3` 统一门户页，把现有监控结果组织成一套统一界面

以上路线能够以最低风险完成题目要求，并保留足够的自研亮点。
